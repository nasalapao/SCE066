# Rule สำหรับโปรแกรมเมอร์ ASP.NET Web Forms

## วัตถุประสงค์

เอกสารนี้ใช้กำหนดแนวทางการพัฒนาโปรแกรม `ASP.NET Web Forms` โดยเน้นให้โค้ดอ่านง่าย ดูแลรักษาได้ง่าย รองรับภาษาไทย และลดการเดา logic เองจากระบบเดิม

เอกสารนี้เป็นกติกาหลักที่ต้องยึดในการเขียนโปรแกรมทุกครั้ง ยกเว้นมีการยืนยันเปลี่ยนแปลง requirement เพิ่มเติมจากเจ้าของงาน

## Technology Stack ที่ต้องใช้

ระบบใหม่ต้องยึดตาม stack นี้

1. `.NET Framework 4.8.1`
2. `C#`
3. `Visual Studio 2022`
4. `ASP.NET Web Forms`
5. `HTML`
6. `CSS`
7. `JavaScript`

**Database ที่ใช้งาน:**
- `DB2 AS400`
- `SQL Server` (ถ้ามี)

**กฎการใช้งาน SQL:**
- ห้ามใช้คำสั่ง `DELETE` และ `UPDATE` SQL เด็ดขาดผ่าน power shell, command หรือ script อื่นที่มีผลต่อข้อมูลจริงโดยตรง ถ้าต้องการลบหรือแก้ไขข้อมูล ต้องทำผ่าน logic ของโปรแกรมเท่านั้น และต้องมี transaction rule ชัดเจน

**ข้อกำหนดเพิ่มเติม:**
- ระบบต้องรองรับภาษาไทยอย่างสมบูรณ์

## หลักการสำคัญ (Core Principles)

### 1. ห้ามมโน logic เอง
ห้ามเดา business logic, SQL, table, field mapping, validation rule, การคำนวณ, การเชื่อมต่อฐานข้อมูล, transaction, format ไฟล์ หรือพฤติกรรมของหน้าจอเดิมเองโดยไม่มีข้อมูลรองรับ

ถ้าข้อมูลไม่ครบ ต้องถามกลับก่อนทุกครั้ง เช่น
- หน้าเดิม submit ไปหน้าไหน / ปุ่มไหนทำงานอะไร
- ต้องการเก็บข้อมูลลง table ไหน / SQL เดิมคืออะไร / field จากไฟล์ map ลง column ไหน
- หน้า result ต้องแสดงข้อความอะไร
- ใช้ Master Page หรือไม่ / CSS เดิมใช้ไฟล์ไหน

### 2. ถ้าข้อมูลไม่ครบ ต้องแจ้งสิ่งที่ขาดให้ชัด
เมื่อ requirement ไม่ครบ ต้องสรุปให้ชัดว่าขาดอะไร (เช่น ไฟล์ CSS, ภาพ, SQL, ชื่อ Table, Flow งาน, Encoding) และต้องขอข้อมูลเพิ่มก่อนลงมือทำในส่วนที่เสี่ยงต่อการทำผิดพลาด

### 3. CSS และ UI
ถ้างานเกี่ยวข้องกับหน้าจอ และไม่มีไฟล์ CSS หรือภาพตัวอย่างแนบมา ให้ใช้ CSS หลักของระบบและโครงสร้างหน้าตาเดิมเป็นหลัก ตรวจสอบว่ามี Shared Style หรือไม่ ห้ามสร้างหน้าตาใหม่เองแบบเดาสุ่ม

## มาตรฐานการเขียน ASP.NET Web Forms และ C#

### 1. โครงสร้างและการแยก Method
- รูปแบบหน้าจอต้องเป็น `.aspx`, `.aspx.cs` และ `.designer.cs` (ถ้ามี)
- โค้ดหลังบ้าน (code-behind) ต้องเขียนด้วย `C#` เท่านั้น (ไม่ใช้ MVC, Blazor ฯลฯ ยกเว้นถูกสั่ง)
- **ต้องแยก method ให้ชัดเจน:** 1 method ควรรับผิดชอบงานเดียว (ตัวอย่างเช่น `BindData()`, `ValidateInput()`, `ShowError()`) ห้ามยัดทุกอย่างไว้ใน `Page_Load`
- ชื่อตัวแปรและ method ต้องสื่อความหมาย
- Control บน Web Forms ควรมี prefix ระบุชนิด (เช่น `txt`, `ddl`, `btn`, `lb`, `gv`)

### 2. การจัดการ Error (Try-Catch และ lbError)
- โลจิกที่มีความเสี่ยงต้องครอบด้วย `try-catch` เสมอ (เช่น การอ่านไฟล์, Database, Request)
- ทุกหน้าที่มีการประมวลผล **ต้องมี** `lbError` ไว้แสดงผล:
  - รีเซ็ตค่าเริ่มต้นก่อนเริ่มงาน: `lbError.Visible = false; lbError.Text = string.Empty;`
  - เมื่อ error เกิดขึ้น: `lbError.Text = ex.Message; lbError.Visible = true;`

### 3. ฐานข้อมูล (Database)
- ใช้ class มาตรฐานของระบบเสมอ: `dbConnect` (สำหรับ DB2) และ `dbConnectSQL` (สำหรับ SQL Server)
- การต่อ SQL นิยมใช้ `string.Format(@"...");` ตาม style ของงาน ยกเว้น class รองรับ Parameterized Query
- การทำงานกับ DB2 AS400 ให้ระวังเรื่อง Library/Schema, Data type, การ Padding string และการใช้ตัวพิมพ์เล็ก-ใหญ่

## มาตรฐานการทำงานร่วมกับ AI (ประหยัด Token)

เพื่อควบคุมขอบเขตงาน ลด Token/Limit และป้องกันการแก้ไขเกินความจำเป็น ให้ยึดหลักดังนี้:

### 1. จำกัดขอบเขตไฟล์และการสั่งงาน (Patch Mode)
- **Patch Mode เป็นค่าเริ่มต้น:** AI ต้องแสดงเฉพาะ code ที่จะแก้ ห้าม Rewrite ทั้งไฟล์ ห้าม Refactor ส่วนอื่น หรือเปลี่ยนชื่อ Method หากไม่จำเป็น
- **เจาะจงไฟล์เสมอ:** ห้าม Scan ทั้งโปรเจกต์ หรือรัน Test/Build ทั้ง Solution ให้ระบุเฉพาะไฟล์หรือ Method ที่กำลังโฟกัส

### 2. ตอบสั้น ตรงจุด และห้ามเพิ่ม Feature เอง
- ตอบเฉพาะสาเหตุและแสดงเฉพาะแพตช์โค้ด ไม่ต้องอธิบายทฤษฎียืดยาว (ตอบไม่เกิน 30 บรรทัดถ้าเป็นไปได้)
- ห้าม AI เพิ่ม Feature, UI, Validation, CSS หรือเปลี่ยน Business Rule นอกเหนือจากที่สั่งโดยพลการ
- การส่ง Context ให้ AI ส่งเฉพาะ Error Message, Method ที่พัง และ SQL ที่เกี่ยวข้อง (ไม่ต้องส่ง log ยาวๆ)

### 3. Prompt มาตรฐานสำหรับงานแก้ไข / พัฒนาจอใหม่

**[Template 1: แก้ Error ประหยัด Token]**
```text
โหมดประหยัด token
งาน: แก้ Error [วาง Error Message]
ไฟล์/Method: [ระบุไฟล์และ Method]
ขอบเขต: แก้เฉพาะจุด ห้าม scan project ห้าม refactor ห้ามเพิ่ม logic ใหม่
Output: สาเหตุสั้นๆ และแพตช์โค้ด
```

**[Template 2: พัฒนาจากระบบเดิม]**
```text
โหมดประหยัด token
งาน: พัฒนาหน้า ASP.NET Web Forms
Input: โค้ดระบบเดิมและ SQL 
Output: ไฟล์ .aspx และ .aspx.cs ที่จำเป็น (ใช้ try-catch, lbError, แยก method)
ขอบเขต: ห้ามเดา SQL หรือ UI นอกเหนือจากระบบเดิม
```

## Checklist การทำงานและส่งมอบ

1. ใช้ `ASP.NET Web Forms` และ `C#` เท่านั้น
2. ควบคุม Error ด้วย `try-catch` และแสดงผลผ่าน `lbError` ชัดเจน
3. แยก Method ควบคุมความซ้ำซ้อน
4. เรียก DB ผ่าน `dbConnect` หรือ `dbConnectSQL` บน SQL ที่ไม่มโนเอง
5. ไม่เดา Logic, ไม่เดา CSS หากข้อมูลไม่ครบได้ดำเนินการสอบถามแล้ว
6. ทดสอบการทำงานร่วมกันของ Client/Server และการรองรับภาษาไทย
7. สั่งงาน AI แบบรัดกุม (Patch Mode) โดยไม่มี Feature แฝง

## มาตรฐาน Git Push

เมื่อผู้ใช้สั่งให้ `git push` ให้ทำตามขั้นตอนทันทีโดยไม่ต้องถามย้ำ:
1. `git status`
2. `git add -A`
3. `git commit -m "[อธิบายสิ่งที่ทำสั้นๆ เป็นภาษาไทย]"`
4. `git push`

*หากเจอเหตุผิดปกติ (เช่น Push ไม่ผ่าน, ไฟล์แปลกปลอม) ให้หยุดและแจ้งผู้ใช้ทันที โดยไม่ต้องสอนทฤษฎี Git นอกเหนือจากที่จำเป็น*
